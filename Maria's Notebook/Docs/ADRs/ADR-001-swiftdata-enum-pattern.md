# ADR-001: SwiftData Enum Raw Value Pattern

**Status:** ✅ Accepted
**Date:** 2025-11 (Discovery), 2026-02 (Documented)
**Deciders:** Development Team
**Tags:** `core-data`, `predicate`, `enum`, `breaking-constraint`

**Note:** This ADR was written during the SwiftData era. The app has since migrated to Core Data, but the raw value pattern remains identical — the same `*Raw` storage + computed enum accessor pattern is used on all NSManagedObject subclasses.

## Context

SwiftData's `#Predicate` macro cannot access computed properties generated by property wrappers. This creates a fundamental incompatibility with elegant enum storage patterns.

### The Problem

Initial implementation attempted to use property wrappers for clean enum storage:

```swift
@propertyWrapper
struct RawCodable<T: RawRepresentable> where T.RawValue: Codable {
    var wrappedValue: T

    var rawValue: T.RawValue {
        get { wrappedValue.rawValue }
        set { wrappedValue = T(rawValue: newValue)! }
    }
}

// Desired usage (DOESN'T WORK):
@Model
class WorkModel {
    @RawCodable var status: WorkStatus  // ❌ Predicate can't access this
}

// Predicate failure:
#Predicate<WorkModel> { work in
    work.status == .active  // ❌ Compiler error: cannot reference property wrapper
}
```

This pattern fails because:
1. `#Predicate` macro expands at compile time
2. It can only access stored properties, not computed ones
3. Property wrappers generate computed properties
4. No workaround exists in SwiftData as of iOS 17/macOS 14

### Impact Scope

- **40+ enum properties** across 60 Core Data entities
- **NSPredicate queries** depend on enum filtering
- **Critical models affected:** WorkModel, Student, LessonAssignment, AttendanceRecord

## Decision

Use the **manual raw value pattern** for all enum properties in Core Data models:

```swift
@Model
class WorkModel {
    // Stored property: Raw value (String, Int, etc.)
    var statusRaw: String = WorkStatus.active.rawValue

    // Computed property: Type-safe enum access
    var status: WorkStatus {
        get { WorkStatus(rawValue: statusRaw) ?? .active }
        set { statusRaw = newValue.rawValue }
    }
}

// Predicate usage (WORKS):
#Predicate<WorkModel> { work in
    work.statusRaw == WorkStatus.active.rawValue  // ✅ Accesses stored property
}
```

### Standard Pattern (3-Line Rule)

Every enum in a Core Data model must use this exact pattern:

```swift
// 1. Raw value storage (use enum's rawValue type)
var [propertyName]Raw: [RawValueType] = [EnumType].[defaultCase].rawValue

// 2. Type-safe computed property
var [propertyName]: [EnumType] {
    get { [EnumType](rawValue: [propertyName]Raw) ?? .[defaultCase] }
    set { [propertyName]Raw = newValue.rawValue }
}
```

## Consequences

### Positive

✅ **Predicates work reliably** - Can filter by enum values
✅ **CloudKit compatible** - Raw values sync correctly
✅ **Type-safe API** - Consumers use enum, not strings
✅ **Migration-friendly** - Can change enum cases with migration
✅ **Codable by default** - Raw values encode/decode automatically

### Negative

❌ **Verbose** - 3 lines per enum property (vs 1 line with property wrapper)
❌ **Repetitive** - Pattern must be repeated 40+ times
❌ **Error-prone** - Easy to forget default case in getter
❌ **Not DRY** - Same pattern copied everywhere
❌ **Code generation needed** - Could use Sourcery but adds complexity

### Neutral

⚠️ **Framework limitation** - Not a design choice, but a constraint
⚠️ **Well-documented** - This ADR prevents future attempts to "fix" it

## Alternatives Considered

### 1. Property Wrapper with @Transient
```swift
@Model
class WorkModel {
    var statusRaw: String = "active"

    @Transient
    @RawCodable var status: WorkStatus
}
```
**Rejected:** `@Transient` properties aren't persisted; defeats the purpose.

### 2. Separate Predicate Helper Methods
```swift
extension WorkModel {
    static func predicateForStatus(_ status: WorkStatus) -> Predicate<WorkModel> {
        #Predicate { $0.statusRaw == status.rawValue }
    }
}
```
**Rejected:** Adds abstraction layer; doesn't solve root problem.

### 3. Use Int/String Directly (No Enum)
```swift
@Model
class WorkModel {
    var status: String = "active"  // No enum at all
}
```
**Rejected:** Loses type safety; allows invalid values; harder to refactor.

### 4. Wait for Apple to Fix
**Rejected:** No indication this will change; blocking current development.

## Implementation Notes

### Enum Requirements

Enums used in Core Data models must:
- Conform to `RawRepresentable` (usually via `String` or `Int` raw values)
- Provide a default case for fallback in getter
- Use stable raw values (don't change existing values)

### Migration Strategy

When changing enum cases:
1. Add new raw value (don't remove old ones yet)
2. Create migration to update raw values in database
3. After migration deployed, mark old cases as deprecated
4. Eventually remove deprecated cases

### CloudKit Considerations

- String raw values are CloudKit-compatible
- Int raw values work but are less readable in CloudKit Dashboard
- Bool should use `"true"`/`"false"` strings, not 0/1

### Testing Pattern

```swift
func testEnumRawValuePattern() {
    let work = WorkModel()

    // Set via type-safe enum
    work.status = .completed

    // Raw value updated correctly
    XCTAssertEqual(work.statusRaw, "completed")

    // Predicate filtering works
    let predicate = #Predicate<WorkModel> { $0.statusRaw == "completed" }
    // ... test predicate
}
```
